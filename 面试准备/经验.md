我们最近面试了一位候选人，岗位方向偏 Infra / System Engineering，需要同时覆盖底层实现与架构设计能力。流程两轮：

1. **Coding（in-memory data store）**
2. **System Design（multi-tenant CI/CD platform）**

---

## 第一轮：Coding —— In-Memory Data Store

开场我给了一个比较模糊的提示：“**可能会考到 Python 序列化**”。

很多候选人听到这个提示后，往往会先入为主地以为是 serialization demo 或 cache 设计题，但我们实际想考的更偏工程化抽象。

**题目实际是：**

实现一个 KV Store，支持基本的 `set / get`，并要求能**序列化落盘**，在进程恢复时**重新加载**。

这道题我们在 infra 岗出得比较多，考察点不在算法，而在**工程思维与抽象能力**，核心看：

- 能否把问题拆成清晰的两层：**内存层**（读写路径）与 **持久化层**（落盘与恢复）
- 是否熟悉 Python 对象序列化机制：`pickle`、`json`、以及自定义 `__getstate__` 等
- 是否会主动确认需求边界：
    - 线程安全要不要？
    - 需不需要版本化？
    - TTL 要不要？
    - 文件格式怎么定？
    - 时间是否需要 mock？
    - 错误处理/一致性预期是什么？

候选人写代码前花了比较久在理解题意，这本身没错，但他一直没有及时建立核心抽象：

**“我到底要做的是一个 cache、一个数据库、还是一个可恢复的字典？”**

最终代码是写出来了，逻辑结构也还不错，但由于**时间控制偏弱**，自写的 test case 没有完整跑通，导致测试没过。

这轮结束后，我们的记录大致是：

> 理解偏慢；实现基本正确；测试未跑完；属于 borderline case。
>

通常更优秀的候选人会在前 **3–5 分钟**快速通过反问把范围锁定，比如：

“这个 KV store 需要持久化吗？需要支持多线程吗？要不要考虑 TTL？”

然后在明确范围后，迅速落下一个 `class KVStore` 的骨架（`set/get + serialize/deserialize`），再补测试。即使有 bug，也能体现出结构完整、路径清晰。

## 第二轮：System Design —— Multi-Tenant CI/CD

这题我们用得比较多，因为它能很好地区分“只会背架构图”和“能独立构思系统”的候选人。

**题面如下：**

设计一个多租户 CI/CD 系统：当收到 Git 推送时，通过内部 API 接收仓库信息，解析仓库中 YAML 定义的 workflow，调度并执行其中的作业，并允许用户实时查看运行状态和日志。

候选人的整体思路比较稳，按层次拆解为：

- Webhook 接收层（API Gateway）
- Workflow 解析层（YAML Parser + DAG Builder）
- Job 调度与执行层（Runner Pool）
- 日志与状态流（WebSocket）
- 多租户隔离（Namespace + RBAC）

能看出他有实战经验，对 GitHub Actions、Argo Workflows 等工具也比较了解。

但在深入讨论时，暴露出两个典型问题：

### 1）对多租户模型理解不够深入

他能提到 “namespace 隔离”，但没有进一步展开关键点，比如：

**资源配额（quota）**、**执行限流**、以及如何系统性地防止**跨租户信息泄露**（权限边界、数据/日志隔离等）。

### 2）缺乏系统化的扩展与演进路径

MVP 层面的设计没问题，但我们更想听他如何演化为企业级版本，例如：

- workflow versioning
- template library
- artifact storage
- metrics aggregation

这部分他更多停留在“知道有哪些组件”的层面，没有形成“系统如何持续演进”的整体抽象。

他在问答过程中态度很谦逊，也能跟上追问节奏。面完后我们内部讨论的评价是：

> 思路清晰，但深度不足；像是掌握了组件清单，却没有在脑中形成 orchestrator 级别的全图。
>

## 结果与总结

两天后我们发了拒信。不是因为能力差，而是因为整体与岗位预期不匹配：

- Coding 环节未能快速锁定核心问题与抽象目标
- System Design 缺乏体系化的多租户与演进思维
- 整体更像“经验型工程师”，而非“系统型思考者”

先校验一个前提：这份 log 不是为了证明候选人“不行”，而是为了解释一件更关键的事——**为什么一个有经验的工程师，到了这种岗位，依然可能被判定为 bar 不够高。**

---

## 第一轮：Coding —— In-Memory Data Store

### 题目表面不难，真正考的是工程抽象

开场我给了一个相对模糊的提示：

**“可能会考到 Python serialization。”**

这个提示本身就是一个小测试。因为很多候选人一听到 serialization，就会自动联想到：

- 是不是要手写一个 serializer / deserializer？
- 是不是 LRU cache？
- 是不是对象持久化 demo？
- 是不是数据库简化版？

但这道题真正想看的，并不是 Python API 会不会背，而是候选人能不能快速完成一个关键判断：

> **这题的本体到底是什么？**
>
>
> 是一个“会持久化的字典”？
>
> 是一个“最小可恢复状态系统”？
>
> 还是一个“读写路径 + 持久化路径分离”的 infra 组件？
>

我们给出的核心要求是：

- 支持基本 `set / get`
- 支持**序列化落盘**
- 支持进程恢复后**重新加载状态**

它不考算法，主要考三件事：

### 1）抽象是否干净

比较成熟的候选人通常会很快把问题切成两层：

- **Memory layer**：服务在线读写
- **Persistence layer**：负责 snapshot / reload / durability

一旦这两层分开，后面很多问题都会自然落位：

- `set/get` 在哪一层
- 持久化时机是什么
- 恢复入口放在哪
- 序列化格式是否可替换
- 错误处理如何不污染主路径

但这位候选人在前期花了较长时间“确认自己到底在做什么”，导致迟迟没有把抽象钉下来。

这类犹豫在普通开发里未必是大问题，但在 infra 面试里会非常显眼，因为这恰恰暴露出一个事实：

> 他在解题时仍然偏依赖“看清题目再开始做”，
>
>
> 而不是主动把模糊题面收敛成一个自己可控的工程模型。
>

### 2）需求澄清是否抓住主轴

候选人确实问了一些问题，但没有迅速打中最关键的边界。

这题比较理想的澄清顺序通常是：

- 这是单机进程内的 store 吗？
- 持久化要求是 snapshot 级别，还是每次写都要保证 durability？
- 要不要考虑并发 / thread safety？
- 数据规模大概多大？是否需要增量写盘？
- 是否接受 process crash 后 last-write 丢失？
- 是否要支持 TTL / delete / versioning？

优秀候选人和一般候选人的差异，不在于“问没问问题”，而在于：

> **他问的问题是否能快速收敛设计空间。**
>

这位候选人在澄清时有点平均发力，花了时间，但没有迅速建立最重要的那条主线：

**这是一个 recoverable KV store，不是泛化数据库，也不是 cache trivia。**

### 3）时间分配与完成度

他的代码整体思路其实不差。

基本实现可以看出他具备实际编码能力，也知道如何把功能拼起来。说明他不是那种“完全不会做”的候选人。

问题出在节奏：

- 前期理解题意用时偏长
- 中段实现推进还算稳定
- 后段测试时间被明显挤压
- 最终 test case 没有完整跑通

在 coding 轮里，这会直接影响评价。因为对 infra 岗来说，**“结构完整但测试没跑完”** 往往不会被视作一个小瑕疵，而会被理解为：

- 缺少交付闭环意识
- 对时间盒管理不够强
- 写出了功能，但没完成自证

所以这轮更准确的评价不是“写错了”，而是：

> **抽象建立偏慢，最终完成度不足。**
>

我们的内部印象大致会落在这种区间：

> 理解偏慢；方向基本正确；实现能落地；但测试闭环没有完成。
>
>
> 属于可通过讨论保留，但不足以形成强烈正向信号的 borderline case。
>

## 第二轮：System Design —— Multi-Tenant CI/CD Platform

### 这题不是考你会不会画图，而是考你有没有 orchestration 视角

题面大意是：

设计一个**多租户 CI/CD 平台**。当收到 Git push 后，系统通过内部 API 获取仓库信息，解析 YAML workflow，调度并执行 jobs，并允许用户实时查看执行状态与日志。

这道题看上去像“GitHub Actions / Argo / Jenkins”的变体，但真正的区分度在于：

> 你是在“罗列组件”，
>
>
> 还是在“定义一个多租户调度与执行系统”。
>

候选人的开局其实是稳的。

他能比较自然地拆出几层：

- Webhook / API ingress
- Workflow parser
- DAG builder
- Job scheduler
- Runner execution pool
- Log streaming / status tracking
- Namespace / RBAC

这说明他不是没有经验，也不是只会背一张空泛大图。

他对 GitHub Actions、Argo Workflows 等系统明显有一定接触，很多术语和分层都不陌生。

但往 deeper dive 走时，问题开始暴露。

---

## 主要问题一：对 “multi-tenant” 的理解停留在表层隔离

候选人能提到：

- namespace 隔离
- RBAC
- 不同 tenant 分开管理

这些当然是对的，但它们还只是**“有租户概念”**，不等于真正理解了**多租户系统设计**。

对于这类题，面试官其实更想听的是下面这些更底层的问题：

### 1）资源公平性怎么定义？

多租户最核心的问题之一不是“分开”，而是“竞争时怎么办”。

例如：

- 一个大客户瞬间提交 10,000 个 jobs，是否会把 runner pool 打满？
- 小租户如何避免被 noisy neighbor 饿死？
- scheduler 是 strict fairness、weighted fairness，还是 priority + quota 混合？
- 是否要按 tenant / repo / workflow 维度做并发上限？

这位候选人没有把这些问题系统性展开。

所以听起来更像是“我知道不同租户要隔离”，而不是“我知道多租户的本质是**共享资源下的公平、保护与可控退化**”。

### 2）数据与控制面的隔离边界不够清楚

多租户平台里，最容易出事的往往不是 compute，而是元数据和日志：

- workflow definition 放在哪里？
- job metadata 是否按 tenant 分库 / 分区？
- logs 是否存在跨租户读取风险？
- artifact URL 是否可能被猜测或越权访问？
- 调试接口、管理员接口、重试接口，权限边界如何收口？

候选人提到了权限和 namespace，但没有进一步形成一个完整的隔离框架。

这会让人担心：**当系统进入真实生产环境时，他可能知道要“做隔离”，但未必能提前识别隔离最脆弱的点。**

### 3）调度器没有成为设计中心

真正好的回答通常会逐渐把设计重心收敛到 scheduler / orchestrator：

- queue model 怎么设计
- DAG ready-state 如何推进
- retry / backoff / timeout 如何处理
- failed runner 的 lease reclaim 怎么做
- 幂等性靠什么保证
- control plane 与 execution plane 如何解耦

而这位候选人的描述，更多还是“有 parser、有 scheduler、有 runner”。

也就是说组件都在，但还没有真正把**执行控制流**讲成整个系统的骨架。

这就是为什么我们会觉得：

> 他像是在讲一个 CI/CD 系统有哪些模块，
>
>
> 但还没有完全站到 orchestrator 的视角上去思考这个系统。
>

---

## 主要问题二：设计有 MVP 感，但缺少清晰的演进路线

候选人第一版设计并不差。

如果把题目理解成“先做一个能跑起来的内部平台”，他的回答是基本合格的。

但这轮面试对于 senior / infra 候选人，真正要拉开差距的，往往是第二层问题：

> **这个系统从 0 到 1 可以这样做，那从 1 到 10 呢？**
>
>
> **从内部工具到企业级平台，它会沿着什么轴演进？**
>

这位候选人在这里显得偏弱。

我们期待听到的通常包括：

- **workflow versioning**：同一 repo 的 workflow 变更如何追踪、回滚、复现
- **template library**：常见 build/test/deploy 流程如何模板化
- **artifact management**：构建产物如何保存、签名、分发、回溯
- **metrics & observability**：tenant 级吞吐、失败率、等待时间、queue depth
- **policy layer**：哪些 workflow 可执行哪些 runner，哪些 secret 可暴露给哪些任务
- **cost attribution / chargeback**：不同 tenant 的资源消耗如何统计与计费
- **security hardening**：untrusted code execution、sandbox、ephemeral runner、secret scoping

候选人对这些点并非完全不知道，但大多是“知道这个组件存在”，而不是把它们串成一条系统演化路径。

这会给面试官一种感觉：

> 他有经验，也接触过这类系统；
>
>
> 但更多是站在“使用/接入/局部维护”的视角，
>
> 而不是站在“平台拥有者”的视角。
>

---

## 面试过程中的整体观感

候选人的一个优点是：**沟通风格不错。**

他并不强辩，也不会在被追问时明显慌乱。

面试官往下挖时，他基本能跟上，也能承认自己没有 fully thought through 的地方。

这是加分项，因为说明他是可以合作、可以被 drive deeper 的人。

但问题在于，到了这个级别的岗位，我们通常希望看到的不只是“配合度好”，而是更强的主动性：

- 主动定义系统边界
- 主动识别 trade-off
- 主动把零散组件抽成一个总控模型
- 主动说明未来三步如何演进

在这点上，他给人的感觉还是偏“经验型工程师”：

- 知道很多部件
- 做过不少实际工作
- 局部判断不差
- 但当题目要求他从上到下组织一个完整系统时，那个“主心骨”还不够强

## 最终结果

两天后发了拒信。

不是因为候选人能力差，甚至也不是因为某一轮“彻底做砸了”。

真正的问题是：**两轮都没有给出足够强的正向信号，去证明他已经达到了这个岗位要的系统性 bar。**

更具体一点说：

### Coding 轮的问题

- 进入核心抽象偏慢
- 没有迅速把题目钉成“recoverable KV store”
- 实现方向基本对，但测试闭环没完成
- 工程感有，但交付感不够强

### Design 轮的问题

- 能搭出组件框架，但设计中心不够稳定
- 对 multi-tenant 的理解停留在 namespace / RBAC 层
- 对 fairness、quota、isolation boundary、scheduler semantics 挖得不够深
- 缺少从 MVP 到 enterprise platform 的演进叙事
